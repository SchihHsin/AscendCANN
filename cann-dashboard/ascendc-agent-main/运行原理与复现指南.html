运行原理与复现指南

一个用 Claude Code Agent 全自动批量开发 + 评测昇腾 NPU 算子的系统。本文拆解它"怎么跑起来"，并重点说明为什么它强依赖 Linux + 昇腾 NPU 硬件，以及在没有 NPU 的情况下能复现到什么程度。

1它到底是什么

不是一个"算子库"，而是一条"让 AI 当算子开发工程师"的自动化流水线。

核心思路：把 35+ 个待开发的昇腾算子写进一份目录（YAML），然后 orchestrator（编排器）逐个为每个算子启动一个独立的 Claude Code 会话，让 AI 在装好 CANN 工具链的 Docker 容器里走完"设计→实现→编译→测试→调优→文档"6 个阶段，全过程留痕成开发日志；再并行启动评估会话给每条日志打分（开发者体验 DX），最后可把报告上传华为云 OBS。

两条主流水线

  • 开发流水线（串行）：逐个算子开发，避免抢占同一张 NPU 卡 / 编译资源
  • 评估流水线（并行）：开发日志互不依赖，用线程池同时打分

产出物

  • logs/batch_{时间戳}/*.md — 每个算子的开发日志
  • *-evaluation.md / *.json — 评估报告与量化指标
  • meta.json — 批次元数据（成功率等）
🔗
它和 cann-dashboard 看板的关系：

这个系统是看板里"算子编程 Ascend C"评分、Agentic KPI（开发成功率/用例通过率/token 消耗/耗时）、客户痛点条目的数据生产端。看板展示的数字，本质来自这里 evaluator 的 5 维度评分聚合。

开发会话的 6 个强制阶段

每个算子的 Claude 会话被 CLAUDE.md "原则 0" 约束，必须一口气走完，中途不停：

# 由 /ascendc-kernel-develop-workflow 引导
方案设计 → 算子实现 → 构建测试 → 问题处理 → 结果总结 → 文档编写
  (docs/)   (ops/代码)  (L0~L3用例)  (修编译/运行)  (回写plan)  (README)

3一次完整运行发生了什么

入口：python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1

1
读配置、建批次

config_loader 从 operator_catalog.yaml 按 --batch / --hardware / --limit 选出算子列表；生成批次 ID YYYYMMDD_HHMMSS，建 logs/batch_{id}/。

2
串行开发每个算子

orchestrator 逐个调 develop_single_operator() → claude_runner 用 subprocess.run(['claude','--print', prompt]) 启一个独立 Claude Code 会话。每个会话拿到的 prompt 指明算子规格 + 强制使用 kernel-develop-workflow / docker / debugging 等 skill。超时 30 分钟/个。

3
会话在 Docker + NPU 里真编译真测试

会话内所有构建/测试命令通过 ./env_setup.sh "命令" 进入容器 cann_container，在挂载了真实昇腾 NPU 设备的环境里跑 Level 0~3 用例。这一步是 NPU 硬依赖的关键。

4
日志落地（Stop Hook）

会话结束时 .claude/hooks/session_stop_handle.py 把整段对话 JSONL 转成 markdown 开发日志，直接写进 logs/batch_{id}/。失败的算子会自动重试 1 次；若检测到 API 429 限流，先等 60 秒。

5
并行评估打分

ThreadPoolExecutor 给每条开发日志再启一个 evaluator 会话，按 5 维度（感知学习/设计实现/编译/调测/调优）输出评分 JSON + *-evaluation.md。这一步只读日志、不碰 NPU。

6
（可选）上传 OBS

加 --upload-obs 时，obs_uploader 把整个批次目录推到华为云对象存储，前缀 cann_exp/static/prod/agent_operator，供看板/外部消费。

常用命令

# 列出可用批次
python3 evaluation_system/orchestrator.py --list-batches

# 完整流程：开发 + 评估（推荐）
python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1

# 只先跑通一个算子做验证
python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1 --limit 1

# 按硬件类筛选 + 上传 OBS
python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1 --hardware Vector --upload-obs

# 只评估已完成的开发日志
python3 evaluation_system/orchestrator.py --evaluate batch_20260115_094732

4NPU 容器机制（env_setup.sh）

这是整个系统"接触硬件"的唯一入口，也是它无法在 Mac 上原样跑的根因。

env_setup.sh 默认 NPU 模式，把命令送进基于 CANN 工具链的容器执行。NPU 模式下它会把宿主机的昇腾设备节点和驱动直接透传给容器：

# 设备节点（昇腾内核驱动创建，仅 Linux + 真实 NPU 主机才有）
--device /dev/davinci1          # NPU 计算设备
--device /dev/davinci_manager   # 设备管理
--device /dev/devmm_svm         # 共享虚拟内存
--device /dev/hisi_hdc          # Host-Device 通信

# 驱动 / 工具透传
-v /usr/local/dcmi              -v /usr/local/bin/npu-smi
-v /usr/local/Ascend/driver/lib64/
-v /usr/local/Ascend/driver/version.info
-v /etc/ascend_install.info

# 项目目录挂进容器
-v ${PROJECT_ROOT}:/ascendc-agent  -w /ascendc-agent

并行多卡开发

orchestrator 可通过环境变量 ASCEND_DEVICE_ID 把不同算子分配到不同 NPU 卡：脚本据此设 ASCEND_VISIBLE_DEVICES 并生成独立容器名 cann_container_device_{N}（N=0~3），互不干扰。

# 镜像（aarch64 / 昇腾 910B / openEuler）
DOCKER_IMAGE = ascendc-optest:latest   # 由本仓 Dockerfile 构建
FROM swr.cn-south-1.myhuaweicloud.com/ascendhub/cann:8.5.0-910b-openeuler24.03-py3.11
💡
CPU 模式存在，但不是"等价替身"。

./env_setup.sh --mode cpu 会跳过 NPU 设备挂载，用 Ascend C 的 CPU 孪生（CPU 仿真）做有限的功能验证。但 Level 1~3 真实精度/性能测试、npu-smi、性能 profiling 都拿不到，开发成功率/用例通过率这类指标也就无法真实产生。

5复现限制 ⚠️（为什么 Mac 上跑不起来）

你当时想复现卡住，就卡在这一层——它有几个"硬"前提。

前提必需性说明 / 为什么
Linux 宿主机 硬性 /dev/davinci* 这些设备节点由昇腾内核驱动在 Linux 上创建。macOS / Windows 内核里根本不存在；Docker Desktop 跑在轻量虚拟机里，也无法把宿主的 NPU 透传进去。这是 Mac 复现失败的第一道墙。
昇腾 NPU 物理卡 硬性 镜像针对 910B。要真编译真测试，必须有插了昇腾卡的服务器（如 Atlas 800/900 训练服务器）。云上可租昇腾算力实例。
NPU 驱动 + 固件 硬性 宿主机要先装好 Ascend driver/firmware，能跑 npu-smi info，且存在 /usr/local/Ascend/driver、/usr/local/dcmi、/etc/ascend_install.info —— 容器靠透传它们才能看到卡。
aarch64（ARM）架构 注意 基础镜像是 openeuler24.03 ARM 版。x86 昇腾环境需换成对应 x86 的 CANN 镜像 tag，否则镜像拉不动 / 跑不起来。
Docker 硬性 能拉华为云 SWR 基础镜像（或本地已缓存），并用本仓 Dockerfile 构建出 ascendc-optest:latest。
CANN 工具链 镜像内置 CANN 8.5.0 已封在镜像里（set_env.sh + ascend-toolkit），一般无需另装。
Claude Code CLI + Token 硬性 真正"写算子的人"是 Claude。宿主要装 claude CLI，并配置 ANTHROPIC_AUTH_TOKENS（逗号分隔可多 token 轮换抗限流）。无 token = 流水线空转。
Python 3.10+ / PyYAML 硬性 跑 orchestrator 本体。
华为云 OBS（AK/SK） 可选 仅当要 --upload-obs 时需要 esdk-obs-python + 凭证；不传就完全用不到。
🚫
一句话结论：

这个仓是原作者在昇腾 NPU 服务器（Linux + 910B）上跑的。在你的 macOS 上，你能读懂全部代码、能跑 orchestrator 的"空架子"逻辑，但到第 3 步进容器编译测试就会失败——因为没有 /dev/davinci* 和 npu-smi。这正是当时复现卡住的原因。

6复现步骤（在合适环境上）

分两种场景：A=有昇腾服务器（完整复现）；B=只有 Mac/普通机器（受限验证）。

场景 A · 昇腾 NPU 服务器（完整复现）

# 0. 前置：确认能看到卡
npu-smi info

# 1. 装依赖 + Claude CLI
pip3 install pyyaml
npm i -g @anthropic-ai/claude-code        # 或官方安装方式
export ANTHROPIC_AUTH_TOKENS="sk-xxx,sk-yyy"   # 可多 token 轮换

# 2. 构建容器镜像（首次，env_setup.sh 也会自动构建）
cd ascendc-agent-main
docker build -t ascendc-optest:latest .

# 3. 冒烟测试：进容器确认环境
./env_setup.sh "npu-smi info && cat /usr/local/Ascend/ascend-toolkit/latest/version.cfg"

# 4. 先只跑通一个算子（强烈建议）
python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1 --limit 1

# 5. 跑整批
python3 evaluation_system/orchestrator.py --full-pipeline --batch day_1

场景 B · 没有 NPU（只能验证编排骨架）

  • 能跑：--list-batches、config 加载、prompt 拼装、纯评估（--evaluate 只读已有日志，不碰 NPU）。
  • 能局部跑：./env_setup.sh --mode cpu "..." 做 Ascend C CPU 孪生功能验证（需要能构建 x86/对应架构的 CANN CPU 镜像）。
  • 跑不了：完整 --full-pipeline 的开发阶段——第一个算子进容器测试就会因无 /dev/davinci* 失败。
🧪
想在 Mac 上"看到效果"的折中办法：

用现成的 logs/batch_*/ 开发日志（如果有归档）直接跑 --evaluate，就能复现"评估打分 + 生成报告"这半条流水线，而完全不需要 NPU。开发那半条则建议申请一台昇腾云实例再跑。

7资源与成本估算

主要成本是 Claude token，不是 NPU 机时（机时另算）。

项量级备注
单个 L1 算子开发~20–50 万 tokens（约 $1–3）一个完整 6 阶段会话
全批次（36 算子）约 $40–100+含失败重试；L3 更贵
单算子超时开发 30 分钟 / 评估 1 小时orchestrator 常量 + config
并行度评估默认 2 线程；开发串行--max-workers 可调
抗限流多 token 轮换 + 429 等待 60s + 重试 1 次claude_runner / orchestrator
NPU 卡1–4 张（并行按卡分配）--device-id 0~3
