# Plan 模式和需求分析

## 需求充分性检查

在开始算子开发前，使用默认值处理缺失信息：

### 必需信息（缺失时使用默认值）

| 信息 | 默认值 |
|------|--------|
| **算子名称** | 从需求描述中提取 |
| **数学公式** | 从需求描述推断，参考标准实现 |
| **输入 tensor** | 1D tensor，float32 |
| **输出 tensor** | 与输入相同 shape |
| **数据类型** | float32（可添加 float16） |
| **精度要求** | 相对误差 1e-3（需用户确认） ⭐ |
| **服务器类型** | 通过 `/ascendc-env-check` 动态检测 |
| **精度敏感操作** | 自动识别（累加/exp/log/归一化等） ⭐ |
| **混合精度需求** | 根据算子特征评估 ⭐ |

### 默认配置
- **性能预期**：不预设性能基准，完成功能即可
- **NPU 资源**：动态获取硬件参数
- **特殊约束**：32 字节对齐

## Plan 模式使用规范

### 何时进入 Plan 模式

**强烈推荐**：
- 所有非最简单的算子
- 需要组合多个 API 的算子
- 需要特殊内存管理或优化策略的算子
- 对 Ascend C 实现方案不太确定的算子
- 精度敏感的算子（累加、归一化等） ⭐

**可跳过**：
- 与现有示例几乎完全相同的简单算子

## 精度需求评估 ⭐ NEW

### 询问用户

在制定计划时，必须询问用户关于精度的需求：

- "该算子对精度要求如何？是否需要在中间计算过程中使用 FP32 提升精度？"
- "期望的容差是多少？（默认 rtol=1e-3）"

### 识别精度敏感操作

根据算子特征自动识别是否需要特殊处理：

| 操作类型 | 精度风险 | 推荐方案 |
|---------|---------|---------|
| **累加类**（Reduce/Sum/Mean） | 高 - 精度损失累积 | 使用 FP32 累加器 |
| **exp/log 操作**（Softmax/LogSoftmax） | 高 - 易溢出 | 设计数值稳定算法（如减最大值） |
| **归一化**（LayerNorm/BatchNorm） | 中 - 中间值精度影响大 | 中间值用 FP32 |
| **除法操作**（Div/Normalize） | 中 - 除零风险 | 添加除零保护 |
| **大数减法** | 高 - 灾难性抵消 | 使用数值稳定等价公式 |

### 确定混合精度方案

根据识别结果，确定是否需要混合精度：

1. **无需混合精度**：简单逐元素操作（Add、Mul、Relu 等）
2. **需要混合精度**：
   - 累加类操作：使用 FP32 累加器
   - 归一化操作：中间 mean/variance 用 FP32
   - Softmax 类：中间 exp/sum 用 FP32

### 设定验证标准

明确精度验证标准：

| 数据类型 | 推荐 rtol | 推荐 atol |
|---------|----------|----------|
| FP32 | 1e-5 | 1e-6 |
| FP16 | 1e-3 | 1e-4 |

> 注：最终标准以用户确认为准，如用户未指定则使用推荐值。

## Plan 内容设计

### 1. 算子设计方案

- 数学公式 → Ascend C API 的映射关系
- 数据流设计（输入 → 计算 → 输出）
- 内存管理方案（使用 LocalMemAllocator 等）
- 精度处理方案（哪些中间变量需要 FP32） ⭐

### 2. NPU 性能优化方案

根据数据规模和性能要求选择优化技术：

| 技术 | 原理 | 适用场景 | 性能提升 |
|-----|------|---------|---------|
| **SIMD 矢量计算** | 一次处理多个元素 | 主体计算逻辑 | 10-100x |
| **Tiling 分块** | 大数据分块处理 | 数据量 > 8K 元素 | 充分利用带宽 |
| **双缓冲** | 计算与搬运并行 | 大规模数据 + 高性能要求 | ~2x |
| **流水线同步** | SetFlag/WaitFlag 事件驱动 | 复杂依赖关系 | 减少等待 |

**决策建议**：
- 小数据量（< 8K 元素）：基础实现
- 中等数据量（8K - 1M 元素）：SIMD + Tiling
- 大数据量（> 1M 元素）：SIMD + Tiling + 双缓冲

### 3. 开发步骤

- 按官方模板结构分解任务
- 明确每个文件的开发内容
- 设置验证检查点（每完成一部分验证编译）
- 性能验证点
- 精度验证点 ⭐

## 计划执行

写完 plan 后：
1. 保存至 `./docs/{算子名称}.md`
2. **立即开始执行实现**，无需等待确认

---

## 重要原则

> Ascend C 算子开发不是简单的算法实现，而是要**充分发挥 NPU 硬件性能**。在 Plan 阶段必须深入思考如何利用 AI Core 的并行能力、内存层次结构和流水线优化。

> **精度意识**：对于精度敏感的算子，在 Plan 阶段就应确定混合精度方案，避免后期反复调试。详见 `/ascendc-debugging` 的精度专题部分。 ⭐
