C
00 · 背景
这个看板在解决什么问题？
CANN 体验看板面向产品/体验团队，帮助决策者基于 Agentic 评分、痛点和真实用户声音（VOD），快速定位开发者体验中最需要改进的环节。

👤 看板的使用者

  • CANN 产品 / 体验团队负责人
  • 各角色方向的产品经理
  • 需要向上汇报数据的人

他们需要快速判断：哪里出了问题、有多严重、改进成效如何。

🎯 核心任务

  • 快速定位体验最差的角色 / 场景
  • 了解痛点来源（Agent 评测 vs 真实用户）
  • 追踪改进后的数据变化
  • 为改进优先级排序提供依据

⚠️ 设计出发点

  • 数据分散：评分/痛点/VOD 没有统一视图
  • 缺乏优先级感：不知先改哪里
  • 角色切换体验割裂
  • 缺少整体健康度的直觉判断

Spotify Soundcheck / Backstage

服务健康度自动化评分看板 · 与 CANN 场景卡片结构最接近

多维评分健康热力图改进 Track
最高参考价值
1全景健康热力图 — 所有服务/组件排列成矩阵，颜色即健康状态。CANN 看板的"角色 × 场景"全景视图可借鉴此设计：一眼看出哪个场景是红色。
2聚合指标 + 趋势线 — 每个维度分项打分，并叠加历史趋势。CANN 看板的 Agent 评分可拆为多子项（精度/性能/文档等），同样展示各项变化趋势。
3Check 评分详情 — 每条检查项显示"通过/失败 + 原因说明"。对应 CANN 的痛点条目：每条痛点应附"来源 + 严重程度 + 建议改进"。
4Tracks 长期改进路径 — 将改进项目组织为"路径"，每个 Track 有进度和截止目标。CANN 看板后续可加"改进计划追踪"模块。
核心借鉴：三层结构（总览热力图 → 实体评分卡 → 单项 Check 详情）与 CANN"全景 → 角色 → 场景详情"完全对齐。颜色矩阵让严重问题 3 秒内可见；每个 Check 都有 Next Step 指引，不只是看数据。
来源：Soundcheck 插件页 · Live Demo

DX Core 4 · getdx.com

系统指标 + 感知数据统一评分框架 · 行业最权威 DX 量化体系

系统指标+感知数据行业基准对比
最高参考价值
1DXI 全景仪表板 — 14 个关键驱动因素按维度分组，右侧显示与业务影响的关联。CANN 全景层可类似地将评分 + 痛点数 + VOD 数三列并排。
2标准化指标维度 — 每个指标同时有"系统数据"和"感知调查"两列，颜色区分数据来源。CANN 中 Agent 评分和 VOD 分列展示，互相对照。
3行业 Benchmark 对比 — 你的分数 vs 行业中位数，让"67分"变成有意义的判断。CANN 看板至少应提供角色内横向对比（6个场景间的相对排名）。
核心借鉴：Agent评分（系统指标）+ VOD（感知数据）并用是行业黄金标准，有 800+ 团队数据支撑。每提升 1 分 DX = 节省开发者 13 分钟/周。两者必须同时展示，否则无法形成高置信度改进决策。
来源：DX Core 4 研究论文

Datadog IDP · DORA Metrics + Scorecard

服务目录 + Scorecard 评分卡 + DORA 三合一 · Scorecard 设计与 CANN 场景卡片高度一致

Scorecard 评分卡多维聚合自动数据采集
高度相关
1DORA 指标总览 — 四项指标并排，颜色区分表现等级，同时显示趋势箭头（↑↓）。CANN 全景层的总览指标卡可参照此布局。
2工作流瓶颈分析 — 将整体指标拆解到具体瓶颈步骤，每步显示耗时占比。CANN 场景详情层可类似地把 Agent 评分拆解到具体的"卡点步骤"。
3IDP 服务目录总览 — 左侧导航按服务分类，中间卡片每个服务显示综合健康分 + 当前状态。这是 CANN 角色视图"场景卡片列表"的直接参考。
4Scorecard 评分卡 — 每个规则项显示通过率进度条 + 颜色 + 具体数量。CANN 场景卡片中的评分分项可完全参照此设计：进度条 + 色彩 + 痛点数量气泡。
核心借鉴：Scorecard 的设计语言——进度条 + 颜色 + 数量气泡，三个视觉元素同时传递"完成度、严重程度、证据量"三个维度，且不需要读任何文字。CANN 场景卡片应采用相同的视觉密度。

Apache DevLake

开源 DORA 看板 · 多源数据聚合 + Grafana 可视化

多源聚合开源可导入
局部借鉴
1DORA 框架分类 — 四项指标按"速度 / 稳定性"两个维度归类。CANN 看板的指标也可按"功能易用性 / 性能 / 稳定性"分类展示，而非平铺列表。

Grafana CI/CD DORA 看板

从 CD 流水线提取 DORA 指标 · 部署频率 + 变更失败率可视化实例

可视化多样开源模板
局部借鉴
1数据流架构图 — 从 Argo Workflows + 告警系统 → 统一看板的数据管道。CANN 可绘制类似架构：cannbot-skills + VOD 来源 → 体验看板数据流。
2部署频率时序图 — 时间轴上每次部署标记一个点，颜色区分成功/失败，背景色块表示当前所在等级区间。CANN 评分历史趋势可参照此格式。

核心借鉴：用时序图展示"改进效果"比只展示当前值更有说服力。颜色背景区间（红色/黄色/绿色区域）让数字落在哪个范围一目了然，不需要记阈值。

局部借鉴

02 · 核心洞察
从参考案例提炼的设计原则
每条原则都有行业案例佐证，是本次重构的设计依据，而非主观判断。
🔗

洞察 1：系统指标 + 感知数据必须并用

Agent评分告诉你"流程哪步卡住了"，VOD 告诉你"用户是否真的感知到这个问题"。两者叠加才能形成高置信度改进决策，避免修了技术问题却没解决用户感受。

来源：DX Core 4 研究（800+团队/40,000+开发者）· GitHub DevSat
🎨

洞察 2：颜色编码是优先级传递的最快路径

红/黄/绿的色彩语义编码，让决策者在 3 秒内扫描出最危险的场景，无需逐一读取数字。阈值建议：红 <60、黄 60-80、绿 >80。

来源：Spotify Soundcheck · Datadog Scorecard
📊

洞察 3：展示分布，而非只展示均值

均值会掩盖极端情况。"均分 72"可能隐藏了某次严重失败。用散点图或箱线图展示评测结果的分布，判断"稳定性"而非只看均值。

来源：Apache DevLake DORA Masterclass · Grafana DORA v2 · SquaredUp MTTR 散点图
🧭

洞察 4：每个低分必须有对应的 Next Step

看板的终点是行动，不是观察。为每个痛点条目附"改进建议"，让看板从"报告工具"成为"决策工具"。

来源：LinearB 工程效能看板 · Spotify Soundcheck Tracks
📏

洞察 5：加入参照系，让分数有意义

"算子迁移部署得了 42 分"不如"42 分，低于内部平均 25 分，6 个场景中最低"来得直觉。CANN 至少应提供角色内场景横向排名。

来源：LinearB 2025 Benchmarks · DX Core 4 DXI 指数
🔍

洞察 6：Agent 评测需要"可解释性"

不只给分数，要展示"Agent 在哪步卡住、具体是什么错误"。参照 Stripe API Review 的结构化输出：问题描述 + 发生位置 + 建议修复。

来源：Stripe DX · Twilio AI Agent 评测实践
08 · 用户画像
谁在用这个看板，怎么用
主要受众是内部体验团队，汇报是次要诉求。设计优先服务"日常操作"场景，"汇报模式"作为附加功能。
👩‍💼
体验 PM
主要用户 · 最高频
典型任务 每周/每期评测结束后，扫描整体健康度，找出本期最需要优先改进的 2-3 个场景，推动对应团队跟进。
使用方式 单独查看为主；偶尔在周会上投影。时间有限，需要快速定位问题，不需要读所有数据。
对设计的影响 全景层必须"一眼看出哪里红"；筛选操作要快；痛点要自动排优先级，不能让 PM 自己判断顺序。
👨‍💻
技术方向 Owner
主要用户 · 深度使用
典型任务 聚焦自己负责的角色（如算子开发者），深度查看某个低分场景的 Agent 评测细节和 VOD 原声，判断问题根因。
使用方式 独立查看，看得很深。需要看到足够多的细节：评分子项拆解、每条痛点的具体描述、Agent 运行日志。
对设计的影响 场景详情层要信息充分，不能省略技术细节；支持直接跳转到对应 skill 文档或改进 ticket。
👔
产品总监 / 管理层
次要用户 · 低频 · 汇报场景
典型任务 在季度汇报或专项会议上，了解整体开发者体验健康度，判断资源投入方向是否正确。
使用方式 由 PM 导出摘要后展示，或直接在看板上查看全景层。不会自己操作筛选，需要结论直接可见。
对设计的影响 全景层+总览指标卡需自带结论感；提供一键"导出摘要"功能（PDF/截图），不需要专门的"汇报模式"。
🎯

设计优先级决策

主界面优先服务 Persona 1 + Persona 2（操作密度高、交互深）。Persona 3 的诉求通过全景层的自解释设计（结论直接写在界面上）+ 一键导出摘要满足，不另设"简化版"。

① 产品易用性（三类开发者）
各典型场景下的 Agent 评分和痛点密度，衡量"工具/文档/API 能不能用好"。对应 cannbot-skills 的 ops/ 下各 skill。

⚡ 核心设计判断

这个看板的核心价值不是展示数据，而是帮助决策者找到值得投入资源的改进点。

参照 DX Core 4 研究结论：Agent评分（系统指标）+ VOD（感知数据）两者缺一不可，两者同时展示才能支撑高置信度决策。

各角色 × 各期综合评分的颜色矩阵，是看板的"首屏"。

默认态悬停（显示分数tooltip）点击（进入角色视图）空数据（某角色无评测）

C2 角色选择器

4 种角色 + 芯片型号的组合筛选，是进入专属视角的入口。

未选中已选中（高亮 + 显示综合健康分）禁用（无数据）

C3 场景卡片

单个典型场景的摘要展示：评分 + 痛点数 + VOD 数 + 颜色。

默认（收起）展开（显示详情）严重（红色边框）无数据加载中

C4 评分分项条

场景内 Agent 评分的子项拆解：精度/性能/任务完成各维度进度条。

正常警告（黄）严重（红）无数据（虚线条）

C5 痛点条目

单条痛点：来源标签 + 描述 + 严重程度 + Next Step 链接。

普通高优（红色标记）有 Next Step无 Next Step已解决（灰色划线）

C6 VOD 原声卡

真实用户声音的引用展示：引用文本 + 来源标签 + 日期。

默认展开全文（文字被截断时）无 VOD 数据

C7 总览指标卡

全景层 4 个核心数字：综合评分 / 活跃痛点数 / VOD 条数 / 严重场景数。

正常与上期对比（↑↓ 趋势标记）首期无对比基准

C8 导出摘要按钮

生成本期结构化文字摘要，面向汇报场景。

可用生成中（loading）复制成功（toast 提示）无数据时禁用
11 · 边缘态设计
数据不完整时怎么办
边缘态处理得好，看板才有可信度。每种缺失情况都需要明确的视觉语言，而不是空白或报错。

⏳ 加载中

Agent 评测数据正在获取时。

设计方案 骨架屏（Skeleton）占位，保持布局稳定不跳动；评分区域显示灰色占位条；顶部显示"数据加载中"进度提示。

🈳 无数据

某角色/场景尚未进行评测，或本期暂无数据。

设计方案 场景卡片显示"暂无评测数据"占位态（灰色虚线边框）；不隐藏卡片，让用户知道这个场景存在但未评测；说明"预计下期覆盖"。

❌ 评测失败

Agent 运行过程中出现异常，未能完成评测。

设计方案 用"评测异常"标签替代分数，红色边框提示，附错误类型说明（网络/环境/timeout）；不计入综合均分，避免失败数据拉低其他场景。

🈚 VOD 为空

某场景有 Agent 评分但无 VOD 用户原声数据。

设计方案 VOD 区域显示"暂无用户反馈"，给出置信度提示："当前结论仅基于 Agent 评测，用户感知待验证"——明确告知数据置信度降低，而非假装没问题。

📅 首期无对比

第一次运行，没有历史数据可以对比趋势。

设计方案 趋势标记（↑↓）位置显示"—"，hover 提示"首期数据，下期起展示趋势"；总览指标卡隐藏对比数字，不显示空的"vs 上期"。

⚠️ 数据过时

展示的数据已超出预定更新周期，可能不再准确。

设计方案 顶部 Banner 提示"当前数据更新于 XX 天前，可能不反映最新状态"；数据仍正常展示，不阻断浏览；旁边附"触发重新评测"按钮。
💡

边缘态的设计原则

缺失数据不等于没有信息——"暂无 VOD"本身就是一个有价值的状态，说明这个场景的改进决策置信度较低。边缘态设计的目标是让用户始终知道数据的可信程度，而不是假装完整或直接报错。

12 · 成功指标
怎么知道看板本身做好了
看板是一个产品，它自己也需要被评估。以下是判断看板设计是否达成目标的可验证指标。

⚡ 效率指标

定位最高优先级问题 < 30 秒
从打开看板到说出"本期最需要改进的是 X"，不超过 30 秒。
周期巡检 < 10 分钟
体验 PM 完成 Flow A 全流程的时间不超过 10 分钟。
导出汇报摘要 < 2 分钟
从打开看板到复制摘要内容完成，不超过 2 分钟。

📈 使用质量指标

每期评测后 100% 有人查看
看板不只是数据仓库，每期数据产出后应有实际查看行为。
场景详情展开率 > 60%
打开看板的用户中，有超过 60% 展开了至少一个场景详情。
VOD 区域查看率 > 40%
查看场景详情的用户中，超过 40% 也查看了 VOD 原声。

🎯 业务影响指标

看板评分与实际改进正相关
被标记为"高优先级"的场景，下期评分应有提升，说明看板决策有效。
减少改进决策的讨论时间
引入看板后，"本期改哪里"的对齐时间应明显缩短（可对比引入前）。
VOD + Agent 评分双高的痛点，改进率优先
两类数据都指向同一问题时，该问题被实际解决的比例应更高。
📐

如何在设计评审时用这些指标

在方案评审阶段，可以用效率指标做走查测试："请找到本期最需要改进的场景"——计时，如果超过 30 秒，说明全景层或排序逻辑需要调整。不需要等到上线后才验证，这些指标在原型阶段就能测试。

Q1

VOD 的来源和结构是什么？

cannbot-skills 里只有 Agent 产生的技术痛点，没有真实用户反馈。VOD 从哪来（访谈/问卷/社区工单）？是结构化数据还是原始文字？这决定 VOD 展示区的设计。

Q2

看板展示的是哪一层的评分聚合？

每次跑完一个 skill 产生原始分，看板展示的是单次原始分、多次均值，还是有专门的聚合规则？直接影响总览指标卡的设计。

Q3

AI框架开发者和应用框架开发者的典型场景？

是否对应 pypto-* 系列和 torch-* 系列？还是有单独的场景定义文档？影响看板是否能复用算子开发者的模板。

07 · 设计结论
重构方向总结
6 条核心设计原则（有行业数据支撑）
🎯
问题优先，而非数据优先 — 排序/颜色/强调全以"哪里最差"为导向（Spotify Soundcheck）
🔗
三类证据互相印证 — Agent评分 + 痛点 + VOD原声并列，高置信度决策（DX Core 4）
📐
三层渐进披露 — 全景→角色→场景→详情，每层只展示该层需要的信息（Datadog IDP）
🧭
每个低分有 Next Step — 看板的终点是行动，不是观察（LinearB）
📏
评分需要参照系 — 横向对比让分数有意义（LinearB 2025 Benchmarks）
🔍
Agent 评测需要可解释性 — 不只给分数，要展示 Agent 在哪步卡住（Stripe DX）
