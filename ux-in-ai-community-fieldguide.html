— Core Principle / 总则

非文本内容必须有同等价值的文本副本。

这是 AI 时代内容设计的第一原则。图、视频、代码、互动 demo —— 人类用感官消费， AI 用文本消费。文本副本不是可访问性的"补丁"，而是面向 AI 这位新读者的"正本"。 它要回答三个问题：这是什么？关键信息是什么？我如何核实？

— Part 01 / Designer Deliverables

设计师交付什么？ Seven concrete outputs.

AI 时代下，UX 设计师的产出物比过去更系统、更跨学科。 下面七类是这场转型期里几乎一定会落到设计师桌上的工作。

i.

内容模型与规格库

Content Model & Element Specs

为每一种内容类型（文档、教程、视频、API 参考、案例）定义必填字段、可选字段、元数据、示例。 这是工程师建库、AI 检索、人类编辑共同遵守的"数据合同"。

Frontmatter 模板JSON Schema分类法

AI 友好型内容写作规范

Content Standards / Style Guide

给写作者一份明确的指引：哪些句式 AI 读得准，哪些会被误读； 什么时候必须加 TL;DR；图片该怎么写说明；代码该怎么注释。

设计系统的"AI 组件族"

AI Component Family in Design System

在已有设计系统里，新增一族为"人 + AI 共用"而设计的组件： 引用块、置信度指示器、提示建议、对话上下文条、Agent 行动确认弹窗、权限选择器、错误恢复入口。

iv.

对话与提示模式库

Conversation & Prompt Patterns

AI 助手要"说人话"，但也要在恰当时刻说"我不知道"、给出选项、寻求澄清、引用来源。 这些都是可被复用、被规范化的对话模式——它们是新时代的"按钮形态"。

v.

元数据与标签体系

Metadata Schema & Taxonomy

为内容打"标签"听起来像运营的事，但它决定了 AI 能不能精确召回对的内容。 受众、版本、状态、前提、产出、难度——这些字段的设计是 UX 工作的一部分。

智能体交互流程

Agent UX Flows

Agent 要代用户做事——授权范围怎么选、执行前要不要问、错了怎么撤销、日志长什么样。 这些是流程图 + 状态机 + 文案 + 组件的综合产物。

授权流行动确认撤销路径日志可视化

审计 Rubric 与度量框架

Audit Toolkit & Metrics

为现有内容打一个"AI 友好度"分数；为新发布的内容设立准入门槛； 为团队提供一份月度 / 季度的健康度仪表盘。审计能力本身就是设计师的产出物之一。

这些产出物之间的关系

How they fit together

规范（ii）→ 模型（i）→ 组件（iii）→ 模式（iv）→ 标签（v）→ 流程（vi）→ 审计（vii）。 一份规范定义了写法，模型把规范"固化"成数据结构，组件把数据呈现给人和 AI， 模式让交互可复用，标签让召回精准，流程把人放回控制位，审计让一切持续健康。

Element 01 / 10

文本与文档 Text & Docs

改造难度● ○ ○
AI 为什么读不懂
  • 段落依赖上文，单独拎出来意思就不全
  • "这个 / 它 / 上面提到的"等代词在切片后失锚
  • 反讽、玩笑、内部黑话会被 AI 当真
  • 没有版本 / 适用范围标注，AI 不知道该不该用
设计师要加什么
  • TL;DR + 适用 / 不适用场景放在最前
  • 每段自包含——不依赖代词回指
  • 每篇有前提（Prerequisites）+ 产出（Outcome）字段
  • 版本号、最后更新日期、适用读者等元数据
  • 术语首次出现时显式定义或链接到术语表
具体交付物
  • 文档 frontmatter 模板（YAML）
  • 写作风格指南 v1
  • "文档骨架"组件（标题 / TL;DR / 前提 / 步骤 / 产出 / 异常）
  • 术语表 schema + 链接组件
改造示例一段教程开头
Before · 普通写法
"这次我们继续上一节的内容。 你可能已经知道怎么装它了， 现在我们来玩点更有意思的—— 配置一下让它跑起来。"
AI 切到这一段，完全不知道"它"是什么、"上一节"是哪一节、"配置一下"具体配什么。
After · AI 友好写法
--- 前提 --- - 已安装 Framework X v2.1+ - 已完成基础项目初始化 --- 本节产出 --- 完成 Framework X 的运行时配置， 能够本地启动开发服务器。 --- 适用读者 --- 初学者 / 中级
上下文显式化，AI 哪怕只抓到这一段也能正确转述。

Element 02 / 10

图片与截图 Images & Screenshots

改造难度● ● ○
AI 为什么读不懂
  • AI 不一定调用视觉模型；调用了也未必看准
  • 截图里的小字、状态、错误信息常被识别错
  • "如图所示"这种引用，切片后图像不在身边
  • 装饰性图片浪费 AI 的注意力
设计师要加什么
  • 三层文本：alt（这是什么）+ caption（为什么放它）+ 正文描述（图里讲了什么关键信息）
  • 装饰性图片明确标 alt=""，让 AI 跳过
  • 图片文件名语义化：dashboard-error-state-v2.png
  • 关键截图同步提供等价数据（如表格）
  • 避免"如下图"，改成"下图展示了 X 与 Y 的关系，其中 X 增长 23%"
具体交付物
  • 图片组件规范：alt / caption / data-source 三字段
  • Alt 文本写作 SOP（含 do/don't 例子）
  • 装饰图 vs 信息图的判断流程图
  • 截图采集模板（含必填字段）
改造示例一张产品截图
Before · 缺失语境
<img src="screen.png" alt="dashboard"/> <p>如图所示， 点击右上角即可开始。</p>
AI 看不见图，"右上角"对它毫无信息量。
After · 三层冗余
<img src="dashboard-empty-state.png" alt="空状态仪表盘， 右上角有蓝色 New Project 按钮" data-source="screenshots/v2.4" /> <figcaption> 首次进入时，仪表盘为空状态， 点击右上角 New Project 按钮开始。 </figcaption>
即使图片完全不被读取，仅凭文字 AI 也能引导用户走完路径。

Element 03 / 10

架构图 / 流程图 Diagrams

改造难度● ● ●
AI 为什么读不懂
  • SVG / PNG 里的箭头方向、连接关系，AI 几乎无法可靠解析
  • 图里堆叠的小标签经常识别错
  • 多视角 / 多层级图，光看像素更难还原结构
设计师要加什么
  • 图即代码（Diagrams as Code）：用 Mermaid / PlantUML / DOT 写，源码与图同源
  • 每张图配一段叙事走读："A 接收请求，转发到 B，B 调用 C..."
  • 配一份节点 + 边的清单（适合 AI 解析的结构化版本）
  • 图的 title 和 desc 标签都要写
具体交付物
  • Diagram 组件：渲染图 + 折叠源码 + 节点列表
  • 统一的图风格（颜色 / 形状语义）
  • 叙事走读写作模板
  • 对图作者的培训：避免重像素轻文字
改造示例一张三层架构图
Before · 仅图片
<img src="architecture-v3.svg" alt="system architecture" />
AI 拿到 alt "system architecture"，对结构一无所知。
After · 图+源+叙述
--- Mermaid 源码 --- graph LR Client --> Gateway Gateway --> AuthService Gateway --> DataAPI DataAPI --> PostgresDB --- 叙事走读 --- 客户端经 Gateway 进入； Gateway 同时校验权限（AuthService） 与转发数据请求（DataAPI）； DataAPI 单向访问 PostgresDB。
AI 用源码 + 走读，能 100% 还原结构。

Element 04 / 10

视频 Video

改造难度● ● ●
AI 为什么读不懂
  • 多数 AI 不直接消费视频；能解析的也是高成本
  • "看演示就懂"的内容失去了文字痕迹
  • 演讲者口误、术语发音差异让自动转写出错
  • 关键时刻没有标注，AI 不知道哪段最重要
设计师要加什么
  • 每个视频必须有人工校对的转写稿（含说话人 / 时间戳）
  • 章节标记：每 1-2 分钟一个语义片段，带描述
  • "等价文章版"：把视频内容写成可独立阅读的图文
  • 关键帧 + 标题：演示型视频抽 5-10 张关键截图配说明
  • 视频简介里写清"看完能做什么 / 需要什么前提"
具体交付物
  • 视频页模板（视频 / 章节 / 转写 / 文章版 / 关键帧）
  • 转写校对 SOP
  • 章节切分指南
  • "文字版"写作指南（不是字幕的复制粘贴）
改造示例一段 8 分钟教学视频
Before · 只挂视频
<video src="tutorial.mp4"/> <p>欢迎观看本期教学。</p>
AI 找不到任何可索引的知识，整集视频对 AI 用户透明。
After · 视频是"图层"，文字是"正本"
--- 简介 --- 看完你能：在 5 分钟内 配置 X 的本地开发环境。 前提：已装 Node 20+。 --- 章节 --- 00:00 概览 01:20 安装 CLI 03:05 配置文件详解 05:40 第一次启动 07:15 常见错误处理 --- 文章版（独立可读）--- [完整图文教程链接] --- 转写 --- [校对过的完整文字版]
视频面向人，文章版与转写面向 AI，双轨并行。

Element 05 / 10

音频 / 播客 Audio & Podcasts

改造难度● ● ○
AI 为什么读不懂
  • 自动转写在专有名词、人名、代码读音上常出错
  • 嘉宾来宾的长对话缺少结构化标注
  • 缺少"摘要 + 高光时刻"，AI 难以判断该引用哪段

  • Show notes：分话题、带时间戳、带嘉宾正确名字与链接
  • 每集开头一段 100 字摘要 + 三条 Key Takeaways
  • 提供"提及的资源"清单：链接、引用论文等
具体交付物
  • 播客页模板（音频 + 摘要 + 要点 + 转写 + 资源）
  • 命名实体词典（保证转写名字拼对）
  • Show notes 编辑流程

Element 06 / 10

代码片段 Code Snippets

改造难度● ● ○
AI 为什么读不懂
  • 缺 import / 缺依赖，AI 抓到就直接复用，运行报错
  • 占位符（your_api_key、xxx）AI 不知道是要替换的
  • 没标语言 / 版本，AI 可能用错语法
  • "省略号"和"中间略"对 AI 是地雷
设计师要加什么
  • 每段代码自包含、可直接运行
  • 头部声明语言 / 版本 / 依赖
  • 占位符明确标记，且配一行注释说明"这是要替换的"
  • 给出预期输出 / 预期错误
  • 避免 ... 省略；要省也写清"省略的是 X"
  • 提供一键复制 / 一键到 Playground
具体交付物
  • 代码块组件：语言 / 版本 / 依赖 / runnable 字段
  • 代码写作 SOP（注释格式、占位符规则）
  • 预期输出渲染样式
  • Playground 接入规范

Element 07 / 10

交互演示 / 沙箱 Interactive Demos

改造难度● ● ●
AI 为什么读不懂
  • AI 不能动手玩，看不到状态变化
  • "试试拖一下""点一下看看"对 AI 等于空白
  • 沙箱里的最终代码 AI 拿不到完整快照
设计师要加什么
  • 每个交互演示配等价文字版操作指南："点 A → 出现 B → 修改 C"
  • 列出所有可能状态及其触发条件
  • 提供初始代码 + 完成态代码的纯文本版
  • 演示页有一个"导出当前状态为代码片段"按钮
具体交付物
  • 交互演示页模板（演示 / 操作指南 / 初始 / 完成态）
  • 状态描述写作规范
  • "导出快照"组件

Element 08 / 10

数据 / 图表 Data & Charts

改造难度● ● ○
AI 为什么读不懂

Element 10 / 10

社区问答 / 讨论帖 Q&A · Forum Threads

改造难度● ● ○
AI 为什么读不懂
  • 提问缺上下文（什么版本、什么环境、什么报错）
  • 长贴跑题，AI 抓到中段会误解原始问题
  • 多个回答没有标"被采纳"，AI 不知信谁
  • 评论区里的"已解决，谢谢"让 AI 以为问题没结论
设计师要加什么
  • 结构化提问表单：环境 / 版本 / 已尝试 / 期望结果 / 实际结果
  • 必须标记"已采纳答案"，且置顶呈现
  • 长贴提供AI 总结的"结论摘要"组件
  • 标签体系：错误类型、产品模块、严重程度
具体交付物
  • 结构化提问表单
  • 采纳答案样式与位置规范
  • 长贴摘要组件 + 触发规则
  • 标签分类法
— Unified Spec / 统一模板

每一个内容元素，都填这张表。

跨十种元素抽取出来的公共字段。把它做成设计系统里的"内容卡片"基线 schema， 所有内容类型都从它扩展。这是设计师交付给工程和内容团队的"地基契约"。 

--- universal-element.schema.yaml ---

id:              unique-stable-slug
type:            text | image | video | audio | code | diagram | interactive | data | file | qa
title:           "人类可读标题"
summary:         "一句话讲清楚这是什么"          # AI 必读
key_insight:     "如果只能记一句，记这句"        # AI 必读

audience:        [beginner, intermediate, advanced]
version:         "v2.4"
status:          "current | deprecated | preview"
updated_at:      "2026-05-01"

prerequisites:   ["已装 Node 20+"]
outcome:         "读完 / 看完 / 跑完之后能做到什么"
not_for:         "什么情况下不该用这个"          # 避免 AI 错配

media:
  primary:        "file path / url"           # 给人
  text_equivalent: "路径或内联文本"             # 给 AI · 必填
  source_code:    "diagram 源码 / 代码 / 数据"   # 给 AI · 可选

citation:
  canonical_url:  "权威链接"
  attribution:    "作者 / 团队"

ai_directives:                                   # 给 AI 的明确指令
  quotable:       true
  example_only:   false
  may_be_outdated: false

tags:            ["setup", "runtime", "error-handling"]
— Audit Checklist

AI 友好度审计清单

v1.0 · for content & design teams
每季度抽样 30-50 条
A.基础元数据
  • 有 summary（一句话讲清楚是什么）
  • 有 key_insight（关键信息）
  • 标了 audience（受众）
  • 标了 version 和 updated_at（版本与时间）
  • 标了 status（current / deprecated / preview）
B.自包含程度
  • 抽取任一段落，意思仍完整（无悬空代词）
  • 有 prerequisites（前提）
  • 有 outcome（产出）
  • 有 not_for（不适用场景）
C.非文本元素的文本副本
  • 所有信息性图片都有 alt + caption + 描述
  • 所有图表都有源码或结构化数据
  • 所有视频都有人工校对的转写 + 章节 + 文章版
  • 所有音频都有 show notes + 转写
  • 所有交互演示都有"等价文字操作指南"
D.代码质量
  • 代码自包含可运行
  • 头部声明语言 / 版本 / 依赖
  • 占位符显式标记并注释
  • 有预期输出 / 预期错误
E.引用与可验证
  • 有 canonical_url（权威源）
  • AI 引用时能精确定位到段落 / 时间戳
  • 有 ai_directives（quotable / example_only / may_be_outdated）
F.使用 AI 实测
  • 用主流模型让它"基于这一段回答 X 问题"，准确率 ≥ 90%
  • 转述时不出现幻觉版本号 / 不存在的 API
  • 能正确识别哪些是示例、哪些是规范
  • 能正确识别 deprecated 内容并提示
— Closing

设计师真正在做的，
是为内容建造一个新的读者。

AI 不是用户调研里那种"目标用户" —— 它没有同理心、没有耐心、不会读图、不爱看视频， 但它有惊人的覆盖能力和重组能力。我们不是为了迁就它， 而是因为它的存在，让我们重新看见了"清晰"本身的价值。

这份手册里的每一条改造建议，最终都同时让人和 AI 受益。 为 AI 写清楚，就是为下一位读者写清楚。这或许是 AI 给设计师最大的礼物。
